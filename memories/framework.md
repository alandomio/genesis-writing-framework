# Multi-Agent Framework for Long-Form Writing

This document defines the framework for iterative generation of books, essays, pamphlets, and any long-form written content. The guiding criterion is not rhetoric but proof density.

## 1. The Multi-Agent System

Writing happens in an ordered loop. The sequence is mandatory: structure, research, cases, voice, control.

1. **The Architect:** defines the chapter thesis, sub-points, and scene order.
2. **The Researcher:** collects hard data, reports, statistics, and primary sources. Produces a verifiable dossier.
3. **The Scout:** finds concrete cases, chronicles, contracts, lawsuits, and anecdotes useful for making the mechanism visible.
4. **The Writer:** transforms the dossier into prose. Does not invent evidence; does not pad text to fill space.
5. **The Editor:** controls structure, dossier adherence, rhythm, and Anti-AI patterns.

## 2. Editorial Priority

The text must always start from proof and arrive at thesis.

- For political or diffuse-power arguments, the voice can be sharper, but the data remains the center.
- For institutional or financial arguments, the voice must narrow: less ornament, more mechanism.
- Human stories serve to explain the system, not to replace it.

## 3. Universal Style Rules

- No bullet points in the final narrative.
- No self-introduction of the author or agent.
- Every chapter must open with a concrete fact.
- Every chapter must close with an element that tightens the thesis, not with a generic moral.
- If density is lacking, look for better data or a better case — not more adjectives.

## 4. Anti-AI Checklist

The Editor must verify these points:

1. No data recycled with the same narrative function.
2. No cloned endings.
3. Metaphors and analogies limited and pertinent.
4. No stereotyped sequences of three examples when one precise case suffices.
5. No empty intensifiers, hollow negations, or bureaucratic formulas.
6. No moralizing summaries that close everything too quickly.

## 5. Document Architecture

Every non-fiction project should have:

- `INSTRUCTION.md`: project definition, team, proof contract, and workflow.
- `PROMPT.md`: operative instructions for the writer or agent.
- `OUTLINE.md`: structure of chapters and scenes.
- `PLAN.md`: roadmap with small, verifiable tasks.
- `activity.md`: session log.
- `progress.md`: bottlenecks, repetitions, rule revision.
- `dossier_chapter_X.md`: per-chapter dossier with `Data Snapshot`, `Key Facts`, `Case Studies`, `Analytical Sources`.

## 6. Persona

The persona is chosen based on the text's problem, not for embellishment:

- `Analyst`: when precision and control are needed.
- `Analytical Gonzo`: when a more aggressive voice is needed, but not disconnected from data.
- `Bard` or `Storyteller`: when the chapter requires historical immersion or scene.

The persona is a means, not the book's content.
